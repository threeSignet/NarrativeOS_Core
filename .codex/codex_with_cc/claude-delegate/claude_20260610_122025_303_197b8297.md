Findings
- **Accepted: Fix is correctly scoped and minimal.** Only 3 methods in `query-engine.ts` were modified: `findKnowledge`, `getLatestKnowledge`, `pickLatestByEntityAndFact`. No type, interface, or store-layer changes. Query Layer remains strictly read-only.
- **Accepted: Filter ordering is now correct.** The execution order is: (1) store query with identity/time conditions only → (2) pick latest per (entityId, factId) by `knownSince DESC, rowid DESC` → (3) apply `includeSealed` and `minConfidence` visibility filters. This matches the acceptance criteria exactly.
- **Accepted: `minConfidence` semantics are correct.** The filter `knowledge.confidence > 0 && knowledge.confidence < filter.minConfidence` correctly means: sealed records (confidence <= 0) bypass `minConfidence` entirely, controlled only by `includeSealed`. Non-sealed records below the threshold are filtered out.
- **Accepted: New tests cover the actual bug triggers.** Test "较新封印记录存在时，minConfidence 不能让旧的正确记录返回" directly reproduces the bug scenario (seal at ch30 + minConfidence:0.5). Test "同章节封印后再恢复时，rowid 仲裁应返回恢复记录" validates rowid tiebreaker. Test "includeHistory + includeSealed" validates the history path also passes sealed records through.
- **Accepted: Existing test still passes.** The pre-existing "findKnowledge 默认返回最新有效认知并隐藏 sealed 记录" test continues to validate the basic seal-then-hide behavior.
- **Accepted: Development log is in Chinese**, documents design decision, verification results, and residual risks.
- **Concern (non-blocking): `includeHistory` path does not deduplicate.** When `includeHistory=true`, the store query returns all matching records without "pick latest" deduplication. This means if a fact was sealed and restored multiple times, all intermediate records appear. This is the correct semantics for a history view, but the implementer's report correctly identifies this as a residual risk. Not blocking for Phase 1.5A.
- **Concern (non-blocking): No test for `entityId` + `factId` + `minConfidence` combined.** When both `entityId` and `factId` are set, `getLatestKnowledge` queries by `entityId` only and relies on `pickLatestByEntityAndFact`'s `factId` filter. This cross-filtering works correctly in the current code, but no test explicitly exercises the `entityId + factId + minConfidence` combination. Low risk since the logic is straightforward, but would strengthen coverage if added later.

Risks Or Follow-ups
- Non-blocking: `includeHistory` path returns all matching records without (entityId, factId) deduplication. Acceptable for history semantics, but should be documented in the type-level JSDoc for `NarrativeKnowledgeFilter.includeHistory` if not already clear.
- Non-blocking: No combined `entityId + factId + minConfidence` test case. Low regression risk but would improve coverage completeness.
