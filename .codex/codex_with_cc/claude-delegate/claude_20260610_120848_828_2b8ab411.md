Summary
Fixed the Knowledge latest-state query semantics in Phase 1.5A Query Layer. The core bug was that `minConfidence` was passed to `knowledgeStore.query()` before the "pick latest per (entityId, factId)" logic, causing newer sealed records (confidence=0) to be pre-filtered out at the SQL level and older correct-confidence records to be incorrectly returned as the current state. The fix moves all visibility filtering (includeSealed + minConfidence) to a single post-selection filter in `findKnowledge`, applied after the latest record has been determined. Added 3 new integration tests covering: (1) minConfidence with newer seal, (2) seal-then-restore rowid arbitration, (3) includeHistory + includeSealed.

Changed Files
- src/core/query-engine.ts — Removed minConfidence from store queries in getLatestKnowledge and history path; moved all visibility filtering to findKnowledge outer filter; minConfidence now only applies to non-sealed records (confidence > 0)
- tests/integration/query-engine.test.ts — Added 3 new tests: minConfidence with newer sealed record, seal-then-restore rowid arbitration, includeHistory+includeSealed
- docs/core-development-log.md — Added Phase 1.5A entry documenting the design decision, verification results, and residual risks

Verification
- `npm run typecheck` — Passed, no type errors
- `npm test -- --run tests/integration/query-engine.test.ts` — 7/7 tests passed (4 existing + 3 new)
- `npm test` — 131/131 tests passed across 8 test files, no regressions

Findings
- The root cause was a filter ordering bug: `minConfidence` was applied at the SQL store level (step 1) instead of after latest-selection (step 3). This violated the required query semantics: candidates → pick latest → apply visibility.
- The fix is minimal: 3 methods touched in query-engine.ts (findKnowledge, getLatestKnowledge, pickLatestByEntityAndFact), no interface or type changes.
- The existing test "findKnowledge 默认返回最新有效认知并隐藏 sealed 记录" already covered the basic seal case but not the minConfidence interaction or the restore case. The 3 new tests close these gaps.

Risks Or Follow-ups
- When neither entityId nor factId is set, getLatestKnowledge queries only by source/atChapter, which could return many records. This is pre-existing behavior, not introduced by this change.
- The includeHistory path does not perform "pick latest" deduplication — it returns all records from the store query. If history mode needs per-entity dedup, that requires a separate iteration.
- Phase 2 ThreadResolver / ThreadStore / Retcon commit chain are out of scope and remain as follow-up work.
