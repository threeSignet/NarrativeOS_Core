## 决策记录

- **W8 readCurrentWorldSnapshot**：采用**聚合方案**（写作层枚举实体 + 逐一 `get_context_slice` 聚合），**不新增 Core 全局快照接口**（Phase 7 最小侵入 Core）。若性能不足再议。
- **写作层 vs CLI vs WebUI 边界**：写作层 = 共享功能逻辑（数据/校验/安全/状态机），CLI 与 WebUI 都只是消费它的薄壳。本表所有 W 项均属写作层逻辑，先行完成。
- **W2 重新排序（2026-06-14）**：W2（src/writing/agent/ 桥接层 = AgentCapability 矩阵 + agent-adapter + context-assembly）**推迟到 W13 之后**。依据：设计 §8.4 把桥接层归为 Step B/C（非 Step A）。当前 Agent 的 ReAct 循环只调 Core **工具**（propose_event 等），不调写作层 service 方法；agent-adapter 的"意图→Command→service 分发"路径要到 W13（propose_event 走 CoreBridge）才出现，此前 `AGENT_PERMISSIONS` 矩阵与 adapter **无强制点 = 死代码**。W2 的安全意图已被 **W1（commit_event 工具门控）+ W4（commit/register 必须有审核来源不变式）** 覆盖。因此 Wave 1 改为 W1 → W3 → W4 → W5（数据完整性，均非死代码），W2 在 W13 后回头补齐使其落地即被消费。
- **W5 分支 2 处理方式（2026-06-14）**：§7.11.5 原文分支 2"Core 无记录且 proposal 过期 → commit_failed"**不在 reconcile 实现**。理由：reconcile 时 `author_approved` 既可能是合法待提交，也可能是 Core proposal 跨会话丢失（§7.11.6）的孤儿，二者不可区分；强行判定会**误伤合法待提交 PV**（数据正确性回归）。改为交由 §7.11.6 懒机制：提交时收到 `PROPOSAL_NOT_FOUND` → 标记 PV `expired`。
- **✅ §7.11.6 PROPOSAL_NOT_FOUND → expired 已精确接入（2026-06-14，task #52）**：`commitReviewedProposal` 失败分支按 errorCode 分流——`PROPOSAL_NOT_FOUND` → 标 `expired` + 覆写 humanMessage 指引"重新推演" + isRecoverable=false；其余错误（UNSAFE/STALE/TRANSACTION_FAILED）→ `commit_failed`（可恢复）。状态机已放行 `author_approved → expired`。`tests/writing/core-bridge-audit.test.ts` 原 PROPOSAL_NOT_FOUND 用例改断言 expired，并补 STALE_PROPOSAL → commit_failed 用例。这让 W5 reconcile 分支 2 的"懒处理"语义闭环（用户提交过期 proposal → expired 而非误导性的 commit_failed）。
